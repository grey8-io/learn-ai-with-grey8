# CLAUDE.md — AI Assistant Context

## Project Overview

**Learn AI With Grey8** is a self-driving, open-source AI bootcamp powered by a local LLM tutor. Learners build 15 real projects across 12 progressive phases — from Python basics to deploying AI agents — with zero cloud API keys required. A Socratic AI tutor (running locally via Ollama) guides learners, grades exercises, and provides progressive hints.

Licensed under AGPL-3.0.

### Tech Stack

- **Frontend:** Next.js 14 (App Router, Tailwind CSS, dark-mode default) — `platform/web/`
- **Backend:** FastAPI (async, Python 3.11+) — `tutor/`
- **LLM:** Ollama running llama3.2:3b locally — port 11434
- **Database/Auth:** Supabase (Postgres + GoTrue auth, local Docker) — port 54321 (optional — app works without it)
- **Progress:** Dual-layer persistence — localStorage (anonymous) + Supabase (signed-in) — `lib/progress/`
- **Content Engine:** ACE framework (Click CLI, sync/generate/curate/reflect) — `ace/`
- **Infrastructure:** Docker Compose — `docker/`

### Monorepo Structure


## Architecture

```
Next.js :3000  ──REST──►  FastAPI :8000  ──Ollama API──►  Ollama :11434 (llama3.2)
    │                         │
    └────────────►  Supabase :54321 (Auth + Postgres)  [optional]
    │
    └────────────►  localStorage (anonymous progress)
```

- **Frontend calls backend only.** Next.js never calls Ollama directly — all LLM requests go through FastAPI `/api/` proxy routes.
- **Content is files in git, not database.** The tutor engine reads curriculum content from disk via `curriculum_path` config. The platform serves content through a content API that reads `manifest.json`.
- **`manifest.json` is the source of truth** for the platform. It is auto-generated by `ace sync` from the files on disk. Never edit it manually.
- **URL encoding convention:** lesson IDs like `phase-01/lesson-01` become `phase-01--lesson-01` in URLs (slash replaced with double-dash).
- **Progress persistence is dual-layer.** Anonymous users get localStorage persistence (survives refreshes and restarts). Signed-in users get Supabase database persistence with RLS. On sign-in, localStorage data auto-migrates to the database.
- **Auth is optional.** The middleware allows anonymous access to all `/learn/*` routes. Supabase is only required for multi-user/enterprise setups.

## Progress Persistence

The platform uses a dual-layer progress system that works for both individual learners and enterprise/institutional deployments.

### How It Works

| User State | Storage Backend | Persistence |
|------------|----------------|-------------|
| Anonymous (no Supabase or not signed in) | `localStorage` | Survives refreshes, restarts. Per-browser only. |
| Signed in (Supabase configured) | Supabase Postgres (RLS) | Multi-device, multi-user, enterprise-trackable. |
| Anonymous → signs in | Auto-migration | localStorage data migrates to Supabase, then localStorage is cleared. |

### Key Files

```
platform/web/lib/progress/
  types.ts              ProgressBackend interface, LessonStatus, Submission, QuizResult, ChatMessage, ProgressStats
  local-backend.ts      localStorage implementation of ProgressBackend
  supabase-backend.ts   Supabase implementation of ProgressBackend
  migration.ts          One-time localStorage→Supabase migration on sign-in
  index.ts              Barrel exports

platform/web/components/
  ProgressProvider.tsx   React context — auto-selects backend based on auth state, handles migration
                         Hook: useProgress() → { backend, stats, refreshStats, isReady }
```

### What Gets Persisted

- **Lesson progress:** status (not_started → in_progress → completed), timestamps
- **Exercise submissions:** code, score, test results, rubric feedback (multiple submissions per exercise)
- **Quiz results:** answers with selected options, score (best attempt kept)
- **Chat history:** tutor conversation messages per lesson

### Database Tables (Supabase)

All tables have Row-Level Security (RLS) — users can only access their own data.

- `lesson_progress` — unique constraint on `(user_id, lesson_id)` for upsert
- `submissions` — multiple rows per exercise allowed (submission history)
- `quiz_results` — multiple attempts allowed, best score displayed
- `tutor_messages` — chat history per lesson
- `profiles` — auto-created on signup via database trigger

### Integration Pattern

Pages use the `useProgress()` hook to access the backend:

```typescript
const { backend, stats, refreshStats, isReady } = useProgress();

// Read
const progress = await backend.getLessonProgress(lessonId);
const best = await backend.getBestSubmission(exerciseId);

// Write (then refresh dashboard stats)
await backend.upsertLessonProgress(lessonId, "completed");
await backend.addSubmission({ ... });
refreshStats();
```

## Diagrams in Lessons

Lesson `content.md` files use two types of visuals:

Use ````mermaid` code blocks for flowcharts, architectures, and pipelines. The `MermaidDiagram` component (`components/MermaidDiagram.tsx`) renders these as interactive SVGs with semantic coloring.

**Semantic Color Standard** — nodes are colored by what they represent (keyword matching on node labels), consistent across all diagrams:

| Color | Category | Keywords |
|-------|----------|----------|
| **Blue** | User / entry point | browser, client, user, input, CLI, query, request |
| **Teal** | Processing / middleware | API, FastAPI, server, handler, validate, Nginx |
| **Purple** | AI / LLM | Ollama, model, LLM, generate, embed, agent, train |
| **Amber** | Data / storage | DB, cache, ChromaDB, vector store, file, chunk, index |
| **Emerald** | Output / result | response, result, answer, deploy, display, complete |
| **Rose** | Control / error | error, alert, retry, rate limit, reject, fail |

### ASCII Diagrams (rendered as monospace text)
Use plain ``` code blocks (no language tag) for visuals that don't map to flowcharts: comparison tables, bar charts, scatter plots, code annotations, directory trees, timelines.

### Project READMEs
All 15 project READMEs (`projects/NN-name/README.md`) also use ````mermaid` diagrams following the same semantic color standard. These render on GitHub natively.

### Guidelines for Adding Diagrams
- Add 2-3 diagrams MAX per lesson — only where they genuinely aid understanding
- Place diagrams WITHIN the relevant section, not at the end
- Keep diagrams simple and not overloaded
- Mermaid node labels should contain semantic keywords so colors are assigned correctly

## Lesson Completion

Lesson completion is **evidence-based** — no manual "Mark as Complete" button.

### Completion Requirements

| Phase | Quiz Size | To Complete a Lesson |
|-------|-----------|---------------------|
| **Phases 1-3** (easy) | 5 questions | Pass the quiz (70%+, i.e. 4/5) |
| **Phases 4-7** (medium) | 7 questions | Pass the quiz (70%+, i.e. 5/7) AND pass the exercise (70%+) |
| **Phases 8-12** (hard) | 10 questions | Pass the quiz (70%+, i.e. 7/10) AND pass the exercise (70%+) |

The pass threshold is calculated dynamically from the actual quiz size (`Math.ceil(0.7 * totalQuestions)`) — not hardcoded.

### Skip Path ("I Already Know This")
Students who already know the material can skip the content and go directly to the quiz (and exercise for phases 4-12). Passing proves competence and auto-completes the lesson.

### Auto-Completion Logic
- Triggered automatically after quiz submission (in `quiz/[quizId]/page.tsx`) and exercise submission (in `exercises/[exerciseId]/page.tsx`)
- Lesson page shows a completion tracker with checkboxes for each requirement
- `upsertLessonProgress()` prevents downgrading from `completed` to `in_progress`

### Key Files
- `app/learn/lessons/[lessonId]/page.tsx` — completion tracker UI, skip path links
- `app/learn/quiz/[quizId]/page.tsx` — auto-completes lesson after passing quiz
- `app/learn/exercises/[exerciseId]/page.tsx` — auto-completes lesson after passing exercise
- `components/ChatPanel.tsx` — sliding AI tutor panel (available on both lesson and exercise pages)
- `components/MermaidDiagram.tsx` — renders ````mermaid` code blocks as colored SVG diagrams

## AI Tutor Context Architecture

The Socratic AI tutor uses a **tiered context injection** system that adapts to the active model's context window at runtime. No manual configuration needed — the tutor detects the model and adjusts automatically.

### Runtime Model Detection
- On every chat request, the tutor queries `Ollama /api/show` to detect the model's `context_length`
- Falls back to 2048 tokens for unknown models
- Students can switch models anytime (e.g. `ollama pull mistral:7b`) — the tutor adapts instantly

### Context Tiers

| Layer | Source | ~Tokens | When Injected |
|-------|--------|---------|---------------|
| System prompt | `engine/prompts.py` (Socratic rules) | ~200 | Always |
| Student profile | Frontend sends `StudentProfile` with each request | ~100 | Always |
| Phase roadmap | Manifest (current phase/lesson position) | ~30 | Always |
| Current lesson | `content.md` from disk | ~800 | Always |
| Exercise context | Starter TODOs + test names from exercise dir | ~200 | Exercise page |
| Curriculum index | `curriculum/curriculum_index.json` (pre-generated) | ~500 | Medium+ models (8K+) |
| Chat history | Trimmed from most recent messages | Fills remaining | Always |

### Student Profile
The frontend builds a `StudentProfile` from progress + gamification data and sends it with every chat request:
- Level, XP, streak days
- Lessons completed, current phase
- Strong/weak topics (derived from quiz/exercise scores)
- Hint usage patterns (avg hints per exercise)
- The tutor adapts tone based on profile (e.g. "student rarely uses hints → challenge them more")

### Curriculum Index
Pre-generated during `setup.sh` (Step 9) by `scripts/generate_curriculum_index.py`:
- One-line summary + key concepts for each of the 35 lessons
- Stored at `curriculum/curriculum_index.json`
- Gives the tutor cross-lesson awareness without loading full content
- Regenerate after content changes: `python scripts/generate_curriculum_index.py`

### Key Files
- `tutor/tutor/engine/context.py` — tiered context builder (`build_context()`)
- `tutor/tutor/engine/student_profile.py` — profile-to-text converter
- `tutor/tutor/engine/ollama_client.py` — `get_context_length()` for runtime detection
- `tutor/tutor/engine/prompts.py` — system prompt, grading prompt, hint prompts
- `tutor/tutor/models/schemas.py` — `ChatRequest` with `StudentProfile`
- `scripts/generate_curriculum_index.py` — curriculum index generator
- `curriculum/curriculum_index.json` — pre-generated lesson summaries (auto-generated, do not edit)
- `platform/web/components/ChatPanel.tsx` — builds and sends student profile with each message

## Blocker Prevention (Student Never Gets Stuck)

The platform has a layered system to ensure students are never permanently blocked:

### Layer 1: Detailed Test Failure Messages
- Test runner captures expected vs actual values from pytest assertion errors
- `ExerciseResult` component shows each failed test with:
  - Human-readable test name (e.g. "greet returns correct string" instead of `test_exercise.py::test_greet_returns_correct_string`)
  - Assertion details in monospace font (what was expected, what was received)
  - Failed tests shown first with red cards, passed tests collapsed below

### Layer 2: Smart Failure Guidance
- After a failed submission, contextual tips appear based on the failure pattern:
  - Few tests failing: "Read the test names — they describe what's expected"
  - Many tests failing: "Re-read the lesson content, start with the simplest test"
  - Assertion errors visible: "Check the expected vs actual values above"
  - Always suggests: use Hints or ask the AI Tutor

### Layer 3: Progressive Hints (3 levels)
- Level 1 — Nudge: general direction ("Think about what data structure...")
- Level 2 — Approach: specific concepts without code ("Use a dictionary to map...")
- Level 3 — Near-solution: structure/pseudocode without complete answer

### Layer 4: AI Tutor Chat
- Available on every lesson and exercise page
- Socratic approach — guides without giving answers
- Has full context of the lesson, exercise TODOs, and student's code
- Personalized based on student profile

### Layer 5: Solution Reveal Gate
- After 5 failed attempts, a "Learn from Solution" button appears
- Shows the reference solution with a header: "Study the approach, then try your own version"
- Solution can be hidden/shown — encourages learning, not copying
- Solution served via `/api/content/solution/{exerciseId}` endpoint
- Attempt count tracked per exercise in the progress backend

### Key Files
- `tutor/tutor/grading/runner.py` — `_parse_pytest_output()` extracts assertion details
- `components/ExerciseResult.tsx` — failure display, smart guidance, attempt count
- `app/learn/exercises/[exerciseId]/page.tsx` — solution reveal gate, attempt tracking
- `app/api/content/[...path]/route.ts` — `handleSolutionRequest()` serves solution code

## Adding New Content

1. Create a YAML spec in `ace/specs/` (see existing specs for format)
2. Run `ace generate --spec specs/your-spec.yml`
3. Review generated files, edit as needed
4. Add 2-3 diagrams per lesson using ````mermaid` blocks for flows and plain ``` for text visuals (see Diagrams section)
5. Run `ace curate --path curriculum/phase-NN-slug/` to validate quality
6. `ace sync` auto-runs after generate, but run manually if you edited files by hand
7. Run `make test` to ensure nothing is broken
8. Commit and open PR — CI will run curation + sync check

## Adding New Platform Features

1. Pages go in `platform/web/app/` following Next.js App Router conventions
2. Shared components in `platform/web/components/`
3. API routes in `platform/web/app/api/` — these proxy to the tutor engine
4. Database schema changes: add a migration in `platform/supabase/migrations/`
5. Environment variables for the frontend must be prefixed with `NEXT_PUBLIC_`

## Adding New Tutor Features

1. Create a new router in `tutor/tutor/routers/`
2. Include the router in `tutor/tutor/main.py` via `app.include_router()`
3. Define request/response models in `tutor/tutor/models/schemas.py`
4. Add a corresponding proxy route in `platform/web/app/api/` if the frontend needs access
5. Write async tests with `pytest-asyncio`

## Low-End Hardware Optimization

The platform is designed to run on machines with as little as 4 GB RAM. Hardware-aware tuning is automatic.

### How It Works

1. `setup-ollama.sh` detects system RAM and GPU VRAM
2. Writes a hardware profile to `.ollama_hw_profile` (tier, num_ctx, num_batch, num_gpu)
3. The tutor engine reads this profile at startup (`tutor/tutor/config.py:get_ollama_options()`)
4. Every Ollama API request includes the optimized `options` payload

### Hardware Tiers

| Tier | RAM | num_ctx | num_batch | num_gpu | Model |
|------|-----|---------|-----------|---------|-------|
| **low** | ≤4 GB | 1024 | 128 | 0 (CPU) | `llama3.2:1b` |
| **medium** | 5-8 GB | 2048 | 256 | 0 (CPU) | `llama3.2:3b` |
| **high** | 9-16 GB | 4096 | 512 | auto | `llama3.2:3b` |
| **ultra** | 16+ GB | 8192 | 512 | auto | `llama3.2:3b` or `llama3.1:8b` |

### Key Files

- `local-dev/scripts/setup-ollama.sh` — RAM/VRAM detection, tier selection, writes `.ollama_hw_profile`
- `local-dev/configs/ollama-modelfile.txt` — Modelfile defaults (num_ctx=2048, num_batch=256)
- `tutor/tutor/config.py` — `get_ollama_options()` loads hw profile, `Settings` has `ollama_num_ctx/num_batch/num_gpu`
- `tutor/tutor/engine/ollama_client.py` — sends `options` dict with every generate/chat request
- `.ollama_hw_profile` — auto-generated hardware profile (gitignored)
- `docs/getting-started.md` — Hardware Guide section with tiers, survival guide, model alternatives

### Override Priority

`TUTOR_OLLAMA_NUM_CTX` env var > `.ollama_hw_profile` values > Ollama model defaults
